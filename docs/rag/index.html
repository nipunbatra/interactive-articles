RAG, From Scratch

You already know how language models predict the next token. You already know how kNN finds nearest neighbours. RAG is what happens when you combine the two.

This page walks through every step of retrieval-augmented generation with concrete examples, interactive 2D embeddings, and four domains.

I. What you already know

RAG combines exactly two ideas. Let's make sure we agree on both before wiring them together.

Language Model

Next-token prediction

Given all previous tokens, predict the next one. GPT, LLaMA, Claude — all left-to-right, one token at a time. Knowledge lives entirely in the weights (parametric memory).

k-Nearest Neighbours

Retrieve similar items

Embed items as vectors. Given a query, find the closest points. Pure geometry, no learning at query time.

The key insight. An LM's parametric memory is frozen at training time. If you ask about events after training, or domain-specific facts it never saw, it either hallucinates or says "I don't know." RAG gives the LM an open-book exam — relevant documents retrieved at query time.

III. Embedding documents

First we need a corpus. Pick a domain and see the documents:

An encoder maps each document to a vector. In reality these are high-dimensional (768-d or more), but for intuition we'll use 2D embeddings. 

docs  = split(corpus)       # chunk text
emb   = encoder(docs)       # embed each chunk
index = FAISS(emb)          # build search index

V. Build the prompt

Retrieval meets generation. We concatenate the retrieved documents and the user's question into a single prompt:

Context
Question 
q_emb  = encoder(query)            # embed the query
docs   = index.search(q_emb, k=2)  # retrieve top-2
prompt = concat(docs, query)        # assemble prompt
output = LLM(prompt)                # generate answer

After this step, RAG is done. The retriever has done its job. Everything from here is standard language modelling.

How does the LM use the context?

It doesn't explicitly "decide" to. Self-attention naturally attends to relevant tokens. The attention weight from output token to context token:

Darker cells = higher attention. When generating a word, the model attends most to matching words in the retrieved context.

Instruction tuning matters. A base LM might continue the prompt weirdly instead of answering. An instruction-tuned LM has learned "Question + Context → Answer." The math is still next-token prediction, but the weights are aligned to produce helpful answers.

Three benefits

  • Reduces hallucination. The LM has real text to reference.
  • Up-to-date knowledge. Update the document store, no retraining.
  • Cheaper than fine-tuning. Weights stay frozen, only the index changes.

IX. When RAG fails

RAG is not magic. Four failure modes to know:

1. Retriever fails

Wrong documents retrieved → garbage context → wrong answer. The embedding model's quality is the ceiling.

2. Context dilution

Too many docs (large k) drowns the signal. Attention spreads thin. The relevant doc gets buried.

3. Redundant documents

Five docs saying the same thing wastes context tokens without adding information.

4. No reasoning over docs

The LM often copies from context rather than synthesising. Multi-hop reasoning across documents is hard.

Citations are not proof. Even if the model writes "[Source 1]", it may cite the wrong document or misrepresent what it says. Citations are a learned output style, not a grounding guarantee.

X. What we built

  • RAG = Retrieval + Generation. Nearest neighbours in embedding space → condition the language model.
  • Five steps. Embed → Index → Retrieve → Build prompt → Generate.
  • Better input, not better model. RAG changes the prompt. Instruction tuning changes the behaviour. The underlying math stays the same.
  • Attention is the mechanism. The LM attends to context tokens via self-attention and "copies" relevant facts into the output.
