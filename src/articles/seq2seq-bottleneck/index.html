The Seq2Seq Bottleneck

Every source token squeezes into one fixed-size vector. Short sentences fit fine; long ones overflow. This is the pain point that made attention inevitable — and it's easy to see once you watch the compression happen live.

~7 minDeep Learning · Seq2Seq · Attention Motivation

In 2014, Sutskever, Vinyals, and Le built the first neural machine translation model: an LSTM encoder reads the source sentence, its final hidden state becomes a context vector c, and an LSTM decoder generates the target conditioned on c.

It worked well on short sentences. It collapsed on long ones. Why? Because every source token — every noun, every verb, every article — had to fit into one 512-dim vector. One year later, Bahdanau et al. would fix it with attention. This interactive shows the failure mode.

Why this fails

Think of the context vector as a compressed zip file. A 512-dim float32 vector is ~16 kbit. To translate "The", you need maybe 10 bits. To translate "The quick brown fox jumps over the lazy dog", you need several hundred bits just to keep track of what each word was. At some length, the encoder has to start dropping information.

The one-year fix. Bahdanau et al. 2014: instead of one context vector, keep all the encoder hidden states, and let the decoder attend to them — pick which ones to look at, at each target step. No more bottleneck, no more length penalty. This became attention, which became self-attention, which became the Transformer.

Run the numbers yourself

info_to_pass   =   H(source)   ≈   L · log(V)
capacity(c)   ≈   d · 8 bits

Source length L, vocabulary V. For a 50k-vocab translation task, each token carries ~16 bits. With L=30 and d=512, you're trying to stuff ~480 bits into 4096 bits of capacity — still fine. At L=300, it's 4800 bits into 4096 — impossible.
